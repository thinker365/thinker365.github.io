# 意识

  1. 缺乏质量意识
  2. 质量文化建设空白
  3. 组织缺乏领导力
  4. 团队之间矛盾重重
  5. 团队规模大
  6. 团队责任不清楚
  7. 缺乏有效的目标管理（OKR/KPI等）
  8. 人员招聘、培训流程混乱
  9. 个人职业素质普遍比较低
  10. 个人技术能力不强
  11. 个人软实力弱

# 流程

  1. 照搬流程（机械、苛刻等）
  2. 流程不容易理解
  3. 繁琐的流程：只做加法
  4. 流程没有贯通（反敏捷、反DevOps）
  5. 流程至上
  6. 缺乏有效的流程改进机制
  7. 糟糕的研发效能度量

# 技术

  1. 技术落后（保守）
  2. 技术不成熟（过于激进）
  3. 重复造轮子（开发市场/开源已有的工具）
  4. 研发平台（包括IDE）不好用
  5. 没有或低效的CI/CD流水线
  6. 研发环境不稳定
  7. 各平台（开发、测试、缺陷管理、项目管理等）没衔接好

# 需求

  1. 需求描述不清楚
  2. 业务复杂没有优化（缺乏业务建模/架构设计）
  3. 需求经常变更（存在极大的浪费）
  4. 需求评审流于形式（导致需求遗留问题多）
  5. 需求管理混乱

# 设计

  1. 糟糕的架构设计（系统复杂）
  2. 没有Design for Test
  3. 没有Domain driven design
  4. 缺乏必要的设计文档（如接口定义文档）
  5. 设计评审流于形式（导致设计遗留问题多）

# 质量

  1. 代码质量差（bugly）
  2. 代码可读性差：不规范（命名、注释、风格）
  3. 代码不具有可测试性（含代码复杂）
  4. 糟糕的代码库管理（包括分支、合并等）
  5. 低效的版本构建、集成（如缺乏增量版本构建）
  6. 代码扫描（静态分析）很少做
  7. 代码人工评审流于形式（导致代码遗留问题多）

# 测试

  1. 测试缺少方法（测试分析/设计能力差）
  2. 测试缺少策略（存在测试浪费）
  3. 测试环境不稳定
  4. 自动化测试程度很低
  5. 没有采用探索式测试
  6. 探索式测试和自动化测试没有有机地结合起来

# 部署

  1. 部署或配置自动化程度不高
  2. 缺乏自动监控/人肉运维
  3. 缺乏可观测性
  4. 不自动收集用户反馈
